# 申請・承認Webアプリ 実装に向けた検討事項

`draft.txt` の要件をもとに、実装に向けてさらに具体化・決定が必要な項目を領域ごとに洗い出しました。それぞれについて検討事項（問い）と、選択肢や推奨される方針を記載しています。

## 1. ユーザー・ロール管理
- **ユーザーの識別と属性**:
  - **検討事項**: ユーザーの氏名、所属部署、メールアドレスなどの属性情報をどこまでシステムで管理するか？
  - **推奨案**: 最低限 `id`, `name`, `email`, `role_id`, `department_id` を持ち、部署ごとの承認パターンに対応しやすくする。
- **ロール（権限）の階層化と権限分離**:
  - **検討事項**: 「一般社員」「部門長」「システム管理者」など、ロールの種類とそれぞれができる操作（閲覧のみ、申請可能、承認可能、マスタ管理可能）の定義をどうするか？
  - **推奨案**: ロール（役割）とパーミッション（操作権限）を分離するRBAC（Role-Based Access Control）の設計を導入し、将来の柔軟な権限追加に備える。

## 2. 認証・セキュリティ
- **ログイン機能の方式**:
  - **検討事項**: Cloudflare環境とオンプレ環境で共通の認証基盤（Azure AD, Google WorkspaceなどのIdP）とSAML/OIDC連携を行うか？
  - **推奨案**: 既存の社内IdPがある場合はSSO（OIDC）連携を第一候補とする。アカウント管理の二重化を防げる。
- **「オンプレ用 簡易ログイン機能」の詳細な仕様**:
  - **検討事項**: パスワード認証のみか、特定のIPからのアクセス時にID入力のみでログインさせるなどの省略版か？セキュリティリスクに対する対策はどうするか？
  - **推奨案**: オンプレのイントラネット内からのアクセスのみを想定し、IDとパスワードによるベーシックなDB認証とする。開発・検証環境用と割り切る場合は、IP制限を厳しくする。
- **セッション管理**:
  - **検討事項**: Cloudflare Workers環境におけるスケーラブルなセッション維持をどう実現するか？
  - **推奨案**: ステートレスなJWTを採用し、HTTPOnlyなCookieに保存する方式でWorkerやDB（D1）へのアクセス負荷を下げる。

## 3. 承認フロー・パターン管理
- **承認ルートの複雑性**:
  - **検討事項**: 直列承認（A部長→B本部長）のみか？並列承認（A部門とB部門の両方の承認が揃ってから次へ進む）もあるか？申請金額等の条件に応じた承認者の自動追加（条件分岐）は必要か？
  - **推奨案**: 初期フェーズでは「固定の直列承認ルートの複数パターン（テンプレート）」を作成・選択できるシンプルな形とし、必要に応じて並列・条件分岐へ拡張する。
- **承認パターンの管理主体**:
  - **検討事項**: 承認ルートのテンプレート（パターン）を作成・編集できるのは「システム管理者」のみか、「各部門の管理者」にも開放するか？

## 4. 申請の状態遷移（ステータス）と操作
- **ステータス定義の明確化**:
  - **検討事項**: 申請データが持ちうる状態をどう定義するか？
  - **推奨案**: `draft` (下書き), `pending` (申請中・承認待ち), `approved` (承認済み), `returned` (差し戻し), `rejected` (却下), `canceled` (申請者による取り下げ) などのステータスマスタを定義する。
- **差し戻し（リジェクト）の挙動**:
  - **検討事項**: 差し戻された場合、最初からやり直し（申請者へ戻す）か、一つ前の承認者へ戻すか？
  - **推奨案**: システムをシンプルに保つため、「申請者まで差し戻し、修正後に再申請（または新規コピーして申請）」とする。差し戻し時のコメント入力を必須化する。
- **承認履歴・監査ログ**:
  - **検討事項**: 「誰が、いつ、どの状態で、どういうコメントを残して承認/差し戻しを行ったか」の履歴保存要件。
  - **推奨案**: 申請テーブルとは別に `approval_histories` (承認履歴) テーブルを作成し、全てのアクションを時系列で不変のログとして記録する。

## 5. 通知機能
- **通知のチャネルとタイミング**:
  - **検討事項**: 申請時、承認依頼時、最終承認完了時、差し戻し時に、メールやチャット（Slack, Teams等）への通知は必要か？
  - **推奨案**: 承認の滞留（承認忘れ）を防ぐため、最低限メール通知機能、あるいは社内チャットツールへのWebhook通知機能を実装する。

## 6. システムアーキテクチャ・インフラ
- **「オンプレ切り替え可能なラッパー」の設計**:
  - **検討事項**: データベース（D1）等のCloudflareの機能と、オンプレ側のDB（PostgreSQLやMySQL等）を透過的に扱うための設計をどうするか？
  - **推奨案**: Repositoryパターンを採用。Hono内でDBアクセスのインターフェース（例: `ApplicationRepository`）を定義し、環境変数などに応じて「D1用実装」と「オンプレ用DB実装（Kysely等を使用）」をDI（依存性の注入）で切り替えられるようにする。
- **フロントエンド技術の選定**:
  - **検討事項**: HonoでSSR（JSX等）を行うか、React/Vue等のSPAとしてAPI通信ベースで分離するか？
  - **推奨案**: 複雑なフォーム入力や承認パターンのUIが必要になる可能性が高いため、React等によるSPA構築を推奨。または、シンプルさを優先してHono + HTMXの構成とする。
- **ファイルの添付（ストレージ）要件**:
  - **検討事項**: 申請時に領収書や企画書などのファイル添付が必要か？その場合、Cloudflare R2とオンプレ環境のストレージをどう切り替えるか？
  - **推奨案**: ファイルが必要な場合は、ストレージのラッパー（S3互換APIの利用など）を用意し、R2やオンプレのMinIO等を共通のインターフェースで操作できるようにする。
- **非機能要件**:
  - **検討事項**: バックアップ方針、監査用のアクセスログ、エラー監視（Sentry等の導入）などの運用要件はどうするか？
